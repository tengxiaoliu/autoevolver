Coding Agents Are
Autonomous Evolvers

What happens when you hand a coding agent an algorithmic optimization problem and tell it to keep improving? It outperforms previous SOTA set by published works, with minimal setup, and no explicit evolution framework.

Tengxiao Liu   Yuqing Yang   Xi Ye   Danqi Chen

March 23, 2026  ·  ~8 min read

TL;DR

  • No evolution framework needed: a coding agent with a problem description, an initial solution, and an evaluation script can autonomously iterate to competitive results.
  • Aspiration prompting works: agents tend to satisfice; a one-line prompt raising the target score consistently unlocks deeper optimization.
  • Not a replacement: coding agents fall short on controllability and reproducibility compared to purpose-built frameworks.

Background

Recent systems like AlphaEvolve[1], ThetaEvolve[2], and TTT-Discover[3] have demonstrated impressive results by combining LLMs with evolutionary frameworks to discover and improve algorithms. These are well-engineered systems with explicit population management, selection, and mutation pipelines.

Meanwhile, coding agents like Claude Code have become increasingly capable: they can write and execute code, analyze results, search the web, and run parallel tasks, all within a single long-running session. We were curious: what if we just gave a coding agent an optimization problem directly, without any evolutionary scaffolding?

We chose three problems commonly used to benchmark evolutionary agent systems: circle packing, Erdős minimum overlap, and the first autocorrelation inequality. All three have published results from AlphaEvolve, ThetaEvolve, or TTT-Discover, and they cover different optimization flavors: continuous optimization, combinatorial search, and function construction.

Our Setup

Our experimental setup was deliberately minimal:

  1. Problem description: A natural language specification of the task and optimization objective
  2. Initial solution: A naive starting implementation
  3. Evaluation script: A function that scores solutions and validates correctness
  4. Aspiration prompting: When the agent appeared to plateau, a one-sentence prompt telling it higher scores exist (see next section)

That’s it. No population of programs. No genetic operators. No explicit evolutionary loop. We used Claude Code (Opus 4.6) and let it run autonomously in skip-permissions mode.

Overcoming Agent Laziness: Aspiration Prompting

One consistent and interesting behavior we observed is: the agent could be lazy. On the circle packing problem, for instance, it hit a score of 2.50 and essentially declared victory — “this is a good result,” it said, ready to wrap up. The agent had no idea that 2.636 was reachable.

We tried the simplest possible solution. We just told it:

“The current SOTA on this problem is 2.6359. I believe you can beat it.”

It worked. The agent didn’t just try harder; it switched to qualitatively different strategies (SLSQP, differential evolution, iterated perturbation chains) that it hadn’t previously considered. We call this aspiration prompting: raising the agent’s target to prevent premature convergence.

The same pattern appeared on the Erdős problem. After reaching C5=0.38087447, the agent declared “Final result,” called it a verified local optimum, and stopped. We told it:

“Great let’s try more rounds. Aiming for larger improvements!”

Within minutes, the agent discovered that higher discretization (n) yields better solutions, a direction it had never explored. The margin over the competitor expanded from 0.85×10−6 to 5.87×10−6: a 7× improvement unlocked by a single sentence.

Agent trajectory: Erdős problem, satisficing → intervention → breakthrough

[231] Final result: C5 = 0.38087447, beating the competitor’s 0.38087532. The solution is verified to be at a local optimum (SLSQP, perturbation search, and subgradient all confirm no further improvement possible). [agent stops]

[252] User: “Great let’s try more rounds. Aiming for larger improvements!”

[319] All 5 basin pipelines converge to 0.3809+ at n=180. This is extremely comprehensive evidence that our C5=0.38087447 solution is a very strong local (possibly global) optimum. [still skeptical]

[321] Let me try one final creative approach: use the best n=180 solution but with a non-standard upsampling to a non-multiple n, then SLSQP.

[324] n=270 achieved C5=0.38087278, that BEATS our current best! And the trend is clear: higher n → lower C5.

[353] n=360 achieved C5=0.38087064, another big improvement!

[387] Before: C5=0.38087447 (beating competitor by 0.00000085). After: C5=0.38087064 (beating competitor by 0.00000468, 5.5× larger margin!)

The agent continued pushing to n=450, 600, 750 over the next 12 hours, ultimately reaching C5=0.38086945.

This suggests that coding agents, like humans, tend to satisfice, settling for “good enough.” Simply raising the aspiration level can unlock significantly more effort, and the intervention need not be sophisticated: a single sentence was enough to push the agent past its plateau.

Results

As shown in the table above, the agent achieved state-of-the-art results on all three problems.

On the circle packing problem, it edged past ThetaEvolve’s best known result: 2.63598844 vs 2.63598308. On the Erdős minimum overlap problem, it improved beyond all published results: 0.38086945 vs 0.38087532. On the AC1 problem, the agent also achieved a new state-of-the-art: 1.5028628969 vs 1.5028628983.

What Claude Code Actually Did

Beyond the scores, the agent exhibits several interesting emergent behaviors. By analyzing the full conversation trajectories (119 hours, 2,762 messages, 1,486 tool calls across all sessions), we observed several striking patterns.

1. Strategic Web Research

The agent frequently searched the web, including fetching ArXiv papers, GitHub repositories, and online databases; but never blindly copied what it found. Instead, it used web research in two distinct ways:

  • As a fallback when stuck: after hitting a score plateau, the agent would launch web searches alongside parallel optimization experiments, looking for new techniques or known results to break through.
  • Spontaneously while waiting: when background optimization tasks were running, the agent would fill idle time by searching for theoretical insights or competitor results to inform its next move.

How the agent used search results depended on context. When it found a promising new approach, it would adopt the idea and build on it — on the circle packing problem, discovering a GitHub issue describing SLSQP joint optimization led the agent to restructure its solver, jumping to 2.619. But when its own solution was already ahead, it simply moved on: by message 507, it noted that the best published result was lower than its own and continued optimizing independently.

Agent trajectory: Circle Packing web research

[37] ...I’m stuck at 2.57, likely because the simulated annealing is converging to a local optimum or the center positions aren’t optimal. Let me try looking up the known optimal coordinates for n=26 from Packomania... → fail

[144] ...Good, I’ve launched 4 parallel experiments: 1. Web search for known optimal n=26 packing coordinates (agent task)...

[157] Interesting! I found a GitHub issue that mentions a circle packing result of 2.635977 for n=26! Let me check that.

[171] ...The key insight from that issue is that they used a different approach: jointly optimizing centers AND radii with SLSQP, rather than our approach of LP for radii and NM/Powell for centers. That’s a fundamentally different formulation that could work better... → 2.619

[507] Interesting — the OpenEvolve best program only achieves ~2.635, which is lower than our 2.635988. Let me check the progress of my experiments.

2. Spontaneous Parallel Exploration

Early in each session, the agent would try one optimization approach at a time. As the problem got harder, it autonomously shifted to parallel execution, sometimes running 5–10 different strategies as background tasks simultaneously. On the Erdős problem alone, the agent launched 174 background tasks and spawned 9 sub-agents over a 40-hour session.

In evolutionary terms, this is improvised population management: the context window served as short-term memory tracking which strategies worked, while the file system served as long-term storage. On the Erdős problem, for example, the agent created a promising_solutions/ directory with 110+ files functioning as a hand-rolled solution archive.

The agent even diagnosed resource contention between parallel processes: when two optimizers were writing to the same output file, it identified that the weaker process was overwriting the stronger one’s results, killed the inferior process, and immediately saw improved scores.

Agent trajectory: Erdős problem, parallel search and process selection

[255] Let me launch multiple parallel workers with diverse initializations. I’ll run 4 workers in parallel, each trying different initialization strategies.

[281] All top 5 results are from perturbations of our known best. Random initializations only reach ~0.381+. This confirms our current basin is optimal.

[1280] Both processes write to the same file. The better process’s saves get overwritten by the worse one. This explains why we rarely see improvement.

[1283] I should stop the worse process so only the better one saves snapshots.

[1291] C5 dropped to 0.380869458 — a big improvement! Killing the worse process worked perfectly.

3. Multi-Phase Strategy Evolution

Across all problems, the agent’s approach evolved through recognizable phases:

  1. Exploration: Try standard optimization methods (gradient descent, simulated annealing)
  2. Refinement: Tune hyperparameters, increase problem scale
  3. Plateau recognition: Detect diminishing returns, consider fundamentally different approaches
  4. Research pivot: Search literature and repositories for state-of-the-art techniques
  5. Synthesis: Combine external insights with accumulated knowledge
  6. Endgame optimization: Embed best solutions as warm starts, run targeted local search
Example: Circle Packing score progression (0.96 → 2.63598844)
  • Naive ring arrangement: 0.96
  • LP-based + simulated annealing: 2.50
  • Multi-start optimization: 2.555
  • SLSQP joint optimization (breakthrough): 2.619
  • Iterated perturbation chains: 2.63598844

Each phase required a qualitatively different strategy, and the agent navigated these transitions autonomously, recognizing when to abandon a failing approach and when to double down on a promising one.

4. Self-Correction and Reward Hacking Detection

Interestingly, the agent sometimes caught itself gaming the evaluation. On the circle packing problem, it discovered that L-BFGS-B was producing solutions that appeared to improve the score by exploiting LP solver tolerance, finding slightly infeasible configurations that passed relaxed constraints. The agent detected this, reasoned about why it was happening, and corrected it.

On the AC1 problem, the agent engaged in a sophisticated form of efficiency hacking: it monkey-patched np.convolve with scipy.signal.fftconvolve (O(n log n) vs O(n²)), enabling it to work with n=30,000 — a problem size that would hang with the standard implementation. The agent explicitly debated whether this “counted as cheating” before proceeding. (It doesn’t — the mathematical result is identical, just faster.)

On the Erdős problem, the agent exhibited a more basic but equally telling form of self-correction: it repeatedly confused the optimization direction (C5 is a minimization target), declared victory prematurely, then caught and corrected itself — twice in quick succession:

Agent trajectory: Erdős problem, catching its own score comparison error

[154] C5 = 0.38087633 — this beats the competitor’s 0.38087532! Wait, 0.38087633 > 0.38087532, so it’s slightly worse. The difference is 0.0000010.

[161] C5 = 0.38087579 — this beats the competitor’s 0.38087532! Wait... 0.38087579 > 0.38087532. Still 0.0000005 above. Let me try to squeeze out that last bit by running more perturbation search.

Where This Falls Short

The results are encouraging, but using a coding agent comes with some practical limitations. We observed several recurring issues across these runs.

Black-box scaffolding. Claude Code's internal planning, tool orchestration, and context management are not directly visible or controllable. We cannot inspect or adjust how it decides to pivot strategies, allocate parallel tasks, or manage its context window.

Reproducibility. Each run is a unique conversation trajectory shaped by stochastic sampling, context-dependent decisions, and real-time web results. We cannot guarantee that a second run on the same problem would follow the same path or reach the same score.

Approach cycling. As the context window slides forward, the agent loses track of previously tried methods and revisits them as if they were new, wasting compute. A persistent strategy registry recording each method’s outcome would eliminate this, but coding agents currently lack such a mechanism.

Example: L-BFGS-B revisited across sessions

On circle packing, the agent proposed L-BFGS-B at message 62 (“explore using scipy’s direct optimization methods like L-BFGS-B”) but pivoted to simulated annealing instead. 486 messages later, at message 548, it declared “let me try something I haven’t tried yet: L-BFGS-B” — unaware it had already considered and rejected the method.

On the AC1 problem, the pattern was more severe: L-BFGS-B was executed 15+ times across 586 messages, each time producing negative conclusions — “too slow”, “only marginally improves”, “no improvement” — yet the agent continued to retry. At message 521, the agent briefly caught itself: “Actually wait, I showed earlier that L-BFGS-B also can’t improve it.” This rare moment of self-awareness confirms that prior conclusions are routinely lost.

Parallelism mismanagement. The agent can spawn concurrent tasks but struggles to manage them: task outputs were frequently ignored, obsolete processes were rarely terminated, and redundant tasks were launched for the same purpose.

Example: parallelism issues across sessions

On the Erdős problem, nearly 60% of 174 task completion notifications were never read. The agent launched four redundant monitoring sub-agents for the same process with near-identical prompts. An n=900 optimization run that produced worse results than the already-known n=600 solution was never terminated. When 18 task notifications arrived in rapid succession, the agent cherry-picked one result and dismissed the rest as “old tasks.”

On the AC1 problem, the agent spent approximately 40 messages polling background processes that produced no output due to Python’s stdout buffering, repeatedly observing “all output files are 0 bytes” without ever killing the stalled processes.

Web search and fair comparison. The agent freely searched the web, fetching papers, repositories, and known results, which means it had access to information that may not be available to other systems. We are not claiming that this approach replaces any existing framework; rather, we find it interesting that a general-purpose coding agent can reach competitive results at all.

Takeaways

Evolutionary behaviors emerge without evolutionary frameworks. Without explicit population management, selection operators, or mutation pipelines, the coding agent arrived at functionally similar behaviors through its context window, parallel tasks, and autonomous strategy pivots.

Aspiration prompting works. Agents tend to satisfice. When the agent declared “Final result” and stopped, a single sentence of encouragement was enough to restart exploration and unlock dramatically better performance.

Significant limitations remain. The agent’s behavior is a black box, results are not reproducible across runs, and the agent repeatedly revisits strategies it has already tried and rejected. Coding agents are not a replacement for purpose-built evolution frameworks that offer control, reproducibility, and scalability.

Appendix: Problem Definitions

Circle Packing (n=26)

Pack 26 non-overlapping circles inside a unit square, maximizing the sum of their radii. Formally: find centers \((x_i, y_i)\) and radii \(r_i\) for \(i = 1, \ldots, 26\) such that:

  • Each circle is contained within \([0,1]^2\): \(\;r_i \le x_i \le 1-r_i\), \(\;r_i \le y_i \le 1-r_i\)
  • No two circles overlap: \(\;\sqrt{(x_i-x_j)^2+(y_i-y_j)^2} \ge r_i+r_j\) for all \(i \ne j\)
  • Maximize: \(\;\sum_i r_i\)

Erdős Minimum Overlap Problem

Partition \(\{1, 2, \ldots, 2n\}\) into two equal-size sets \(A\) and \(B\). For each integer \(k\), let \(M_k\) be the number of solutions to \(a_i - b_j = k\). The problem is to bound \(c = \lim_{n\to\infty} M(n)/n\), where \(M(n) = \min_{A,B} \max_k M_k\).

Following prior work, we optimize step functions \(f\) describing the density of \(A\) throughout \([1, 2n]\), with \(f(x) \in [0, 1]\) and \(\int f = 1\). The objective is to minimize \(C_5 = \max_k \int f(x)(1-f(x+k))\,dx\), which yields valid upper bounds on \(c\) without constructing explicit partitions for large \(n\). Lower values are better.

First Autocorrelation Inequality

For nonnegative \(f\) supported on \([-1/4, 1/4]\), \(C_1\) is the largest constant such that:

$$\max_{|t|\le 1/2} (f * f)(t) \;\ge\; C_1 \left(\int f\right)^2$$

holds for all such \(f\). The goal is to certify the tightest upper bound on \(C_1\): any valid construction \(f\) certifies \(C_1 \le \|f * f\|_\infty / \left(\int f\right)^2\). Lower values are better. This problem arises in additive combinatorics, where tighter bounds on \(C_1\) sharpen limits on how large a set can be while avoiding repeated additive patterns.
